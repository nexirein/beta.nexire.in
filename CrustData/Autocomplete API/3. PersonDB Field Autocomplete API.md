# Realtime Filters Autocomplete API

### [ 🚀 Try Now ](/api#tag/auxiliary-api/GET/screener/linkedin_filter/autocomplete)

This endpoint provides autocomplete suggestions for filters, helping users find relevant filter values while building their search queries. You can use these 

## Available Filters
- `region` - Geographic regions and locations
- `industry` - Industry categories
- `title` - Job titles
- `school` - Educational institutions and schools 

## Endpoint

```
GET /screener/linkedin_filter/autocomplete
```

## Request Parameters

| Parameter     | Type    | Description                                                | Required | Default |
|--------------|---------|------------------------------------------------------------| -------- |---------|
| `filter_type`| string  | Type of filter to get suggestions for.<br/>Possible values:<br/>• `region` - Geographic regions and locations<br/>• `industry` - Industry categories<br/>• `title` - Job titles<br/>• `school` - Educational institutions and schools<br/>  | Yes | - |
| `query`      | string  | Search text to get autocomplete suggestions. Example: "san" for "San Francisco" | Yes | - |
| `count`      | integer | Number of suggestions to return (max: 50) | No | 10 |
| `start`      | integer | Starting index for pagination. Use with `count` to paginate through results | No | 0 |

## Filter Types

The `filter_type` parameter accepts the following values:

- `region` - Geographic regions and locations
- `industry` - Industry categories
- `title` - Job titles
- `school` - Educational institutions and schools

- This endpoint does not consume any credits

PersonDB Field Autocomplete API

Copy Whole Auxiliary API
🚀 Try Now 
Get autocomplete suggestions for PersonDB field values, helping you find exact values for filters when searching people profiles. This endpoint supports fuzzy matching, works with both nested and non-nested fields, and allows optional filtering to narrow down suggestions based on other criteria.

Endpoint
POST /screener/persondb/autocomplete

Request Parameters
Parameter	Type	Description	Required	Default
field	string	The field name to get suggestions for (e.g., "region", "current_employers.name")	Yes	-
query	string	Search text to get autocomplete suggestions, empty allowed	Yes	-
limit	integer	Number of suggestions to return (max: 100)	No	20
filters	object	Optional filter criteria to narrow down suggestions (see Filters section)	No	-
Available Fields
You can get autocomplete suggestions for any field available in the PersonDB. See the PersonDB Discovery API documentation for a complete list of available fields.

Field Types
Non-nested fields - Use directly without dot notation Nested fields - Use with dot notation

info
The field parameter accepts the same field names used in the PersonDB Discovery API filters. If a field is not supported for autocomplete, the API will return an error with a list of valid fields.

Explanation of how it works
Under the hood, we power this autocomplete api via fuzzy matching. The field values are allowed have a tolerance of difference in letters (order & different character) to the query you pass in. The amount of tolerance which is allowed is dynamic based on length of the term.

For implementation, we use the AUTO fuzziness of elastic with default settings. More on it can be read about here. We take all the matching field values and sort them in descending order based on the number of documents and return in response.

Using Filters
The filters parameter allows you to narrow down autocomplete suggestions based on additional criteria. This is particularly useful for contextual autocomplete, where you want suggestions that are relevant to other selected filters.

Filter Structure
Filters follow the same structure as the PersonDB Discovery API:

Simple Filter:

{
  "filter_type": "region",
  "type": "in",
  "value": ["United States"]
}

Complex Filter (AND/OR):

{
  "op": "and",
  "conditions": [
    {
      "filter_type": "current_employers.name",
      "type": "in",
      "value": ["Databricks", "Snowflake"]
    },
    {
      "filter_type": "region",
      "type": "in",
      "value": ["United States"]
    }
  ]
}

Nested Field Scoping (Important!)
When both the field parameter and filters operate on the same nested path (e.g., both on current_employers), the filter and autocomplete are combined to ensure they apply to the SAME array element.

Example Use Case: Get titles for people who are CRO at Databricks

Request:

{
  "field": "current_employers.title",
  "query": "cro",
  "filters": {
    "filter_type": "current_employers.name",
    "type": "in",
    "value": ["Databricks"]
  }
}

What this returns: Only titles from jobs where the person currently works at Databricks, ensuring you get "Chief Revenue Officer" from someone who is actually CRO at Databricks, not someone who was CRO at a different company.

Scoping Behavior:

Filter Path	Autocomplete Field	Behavior
current_employers.name	current_employers.title	✅ Same job - Both conditions apply to same employment record
region	current_employers.title	✅ Separate - Region filter + title suggestions (any company)
past_employers.name	current_employers.title	✅ Separate - Past employers filter + current title suggestions
current_employers.name + region	current_employers.title	✅ Document-level - Both filters + title suggestions
Best Practice
When filtering on nested fields, use filters on the same nested path as your autocomplete field to get contextually relevant suggestions. For example, filter by current_employers.name when getting current_employers.title suggestions.

Credit Usage
This endpoint does not consume any credits

Integration with PersonDB Discovery API
The autocomplete API is designed to work seamlessly with the PersonDB Discovery API. Here are recommended workflows:

Get autocomplete suggestions for the field you want to filter:
curl -X POST 'https://api.crustdata.com/screener/persondb/autocomplete' \
--header 'Authorization: Token $authToken' \
--header 'Accept: application/json, text/plain, */*' \
--header 'Content-Type: application/json' \
--data '{
    "field": "region",
    "query": "new york",
    "limit": 5
}'

Response:

{
  "suggestions": [
    "New York, New York, United States",
    "New York City Metropolitan Area",
    "Greater New York City Area",
    "New York, United States",
    "New York County, New York, United States"
  ]
}

Use the exact value in your PersonDB search filter:
curl -X POST 'https://api.crustdata.com/screener/persondb/search' \
--header 'Authorization: Token $authToken' \

        "column": "region",

    "limit": 100

When to Use Autocomplete vs Fuzzy Search
Use Case	Recommended Approach	Example
Need exact matches	Use autocomplete → exact match (=)	Finding all people in "San Francisco Bay Area" exactly
Exploratory search	Use fuzzy operator (.) directly	Finding people with "engineer" anywhere in their title
Multiple specific values	Use autocomplete → in operator	Finding people at specific companies from a list
Partial/flexible matching	Use fuzzy operator (.)	Finding variations of "software developer"
Contextual suggestions	Use autocomplete with filters	Get titles only for selected company/region
Multi-step filtering UI	Use autocomplete with progressive filters	Build filter interface where each step narrows down options
Discovering common values	Use autocomplete with empty query + filters	Find most common titles at a specific company
Notes
The autocomplete uses fuzzy matching, so it's tolerant of typos in your query
Results are ordered by relevance and frequency
Nested fields must use dot notation (e.g., current_employers.name)
The suggestions returned are actual values from the database, ensuring compatibility with exact match filters
Filters are optional - omit the filters parameter for basic autocomplete without filtering
Nested field scoping - When filter and autocomplete field share the same nested path, conditions apply to the same array element
Backward compatible - Existing implementations without filters continue to work unchanged
Filter syntax - Uses the same filter structure as the PersonDB Discovery API for consistency
