# AI Search Pipeline Architecture (Gemini Implementation)

This document details the architecture, logic, and flow of the AI-powered Job Description (JD) search pipeline implemented in Next.js, Supabase `pgvector`, and Google Gemini.

## High-Level Flow

When a user pastes a Job Description (or types a quick natural language prompt) into the `JDSearchModal`, the data flows through a 4-step orchestration pipeline before finally populating the `FilterModal`.

1. **Extraction (LLM)**: `gemini-2.0-flash` extracts structured filter parameters, including the user's selected **`search_mode`**. It is now instructed with "Recruiter Intuition": dropping `max_experience` by default and generating a **50km location radius** (4-8 satellite towns).
2. **Resolution (Vector DB & External API)**: 
    - Technologies, industries, and departments are searched against Supabase PGVector.
    - **Mode-Driven Job Titles**: 
        - If `search_mode` is **Sniper** or **Location Flex**, the system forces a precise **Boolean Strategy** using exact-quote matching (`"title"`) to avoid dilution.
        - If `search_mode` is **Wide Net** or **Title Flex**, it uses the seed fanning-out strategy to include similar roles.
    - **Location Radius Simulation**: The 4-8 satellite towns generated by the LLM are sent to Prospeo's Suggestions API.
        - **Sniper/Title Flex**: strict filtering for city-level only.
        - **Wide/Location Flex**: allows state/regional expansion.
3. **Assembly & Execution**: Results are passed to the **Waterfall Engine**.
    - For **Wide Net**, it executes a 3-pass sequential search: (1) Exact + City, (2) Similar Roles + City, (3) Exact + Nearby Cities.
    - For other modes, it runs a single optimized Prospeo pass.
4. **UI Update**: Filter payload is injected into the modal with improved badges for boolean strategy path visibility and result tier tagging (`EXACT_MATCH`, `SIMILAR_ROLE`, `NEARBY`).

## Technical Components

### 1. Extractor (`lib/ai/extractor.ts`)

Converts messy HTML/Text into a strict JSON schema.

-   **Model**: `gemini-2.5-flash` (Fast, cost-effective, extremely capable at strict JSON generation).
-   **Prompting Strategy**: We provide strict ENUM lists for fields with a small number of valid values (e.g., Seniority, Headcount Ranges, Funding Stages). For massive lists (Tech, Industry, Department), we instruct the LLM to output the *raw strings* exactly as mentioned in the text.
-   **Output Schema**: Returns an object of type `LLMExtractedFilters` with strongly typed structures as JSON.

### 2. Embeddings Provider (`lib/ai/embeddings.ts`)
*   **Provider:** Google Gemini API (`@google/genai`)
*   **Model:** `gemini-embedding-001` configured with `outputDimensionality: 768`
*   **Dimensions:** 768 (suitable for inner product or cosine similarity searches)
*   **Role:** Converts the raw industry, tech, and department jargon extracted by the LLM into 768-dimensional float arrays.
*   **Batching:** Supports batch embedding (100 at a time with 1s delays) for backend index building to respect Gemini's API rate limits.

### 3. Vector Resolver (`lib/ai/vector-resolver.ts`)

Matches raw LLM output against the exact Prospeo Enum lists.

- **Parallelization**: Searches for multiple extracted technologies simultaneously via `Promise.all`.

### 4. Direct Prospeo Suggestions & Dual-Strategy Titles (`lib/prospeo/client.ts`)

For Job Titles, the system dynamically chooses the best path based on **search_mode**:
- **Boolean Strategy (Forced in Sniper/Location Flex):** The system generates a precise `boolean_search` expression with exact quotes (e.g., `"Fleet Manager" OR "Transport Manager"`). This prevents irrelevant LinkedIn headline noise.
- **Include Strategy (For Wide Net/Title Flex):** For roles where the user wants volume, we fan out search seeds to Prospeo and rank suggestions by frequency.

For Locations:
- **Radius Simulation:** Instead of searching just "Kolkata", we search for 15+ satellite hubs (Howrah, Salt Lake, etc.) to simulate a 50km recruiter radius.
- **Strictness**: Sniper and Title Flex modes filter out broad regional suggestions (states/admin areas) to keep focus on the specific city.
- The Suggestions API ensures every string is a verified Prospeo ID, preventing `INVALID_FILTERS` errors.

### 5. Filter Assembler (`lib/ai/filter-assembler.ts`)

Takes all the disparate outputs from Step 1, 3, and 4, and transforms them into `ProspeoFilters`.

- Consolidates exact pgvector matches into `company_technology.include`.
- Resolves departments into `person_department.include`.
- Places Prospeo API location results into `person_location_search.include`.
- Binds direct LLM enum mappings (like `company_headcount_range`) directly into the format context.

### 6. Orchestration Route (`app/api/ai/extract-and-resolve/route.ts`)

The central API route acting as the controller for all the modules above. 
It ensures that Steps 3 and 4 run in parallel. Thus, resolving 10 tech skills, 5 industries, 5 departments, 1 location, and 1 job title all takes the exact same amount of time as resolving just 1 item.

## Infrastructure Initialization

Because Vector Search requires pre-computed target embeddings, there is a one-time build script located at `scripts/build-embedding-index.ts`.

**Purpose**: It iterates over Prospeo's massive arrays (`TECHNOLOGIES`, `INDUSTRIES`, `DEPARTMENTS`), fetches their embeddings via Gemini `text-embedding-004`, and upserts them into the Supabase database.

**Execution**: Must be run via terminal:
\`\`\`bash
npx tsx scripts/build-embedding-index.ts
\`\`\`
If this script is not run, the application degrades gracefully; the UI simply falls back on a generic extraction payload without mapped values, warning the user that the AI index needs to be built.
